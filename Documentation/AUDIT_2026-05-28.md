# SwiftRigControl Hamlib-Parity Audit — 2026-05-28

Comprehensive review of SwiftRigControl against Hamlib reference,
post-v1.1 parity work. Four parallel audits ran:

1. Radio inventory diff
2. Command-implementation divergence
3. Hamlib capability gap analysis
4. Documentation completeness

This document is the synthesis: every finding cross-checked
against actual Hamlib source, hallucinations rejected, real bugs
prioritized into a release plan.

## TL;DR — The unified action list

Tagged for the next two releases.

   - DocC catalog has no articles for them.
   - README's "Supported Radios" table is missing the seven new
     definitions (ID-31, ID-51, ID-52, IC-92D, IC-R30, TH-D75,
     TX-500).
   - Multiple docs (API_REFERENCE.md, USAGE_EXAMPLES.md) still
     reference "post-v2.0" deprecation removals that actually
     shipped in Phase 1.4 of the v1.0.x line.

### Plan for v1.2.0 (next minor release)

3. **Five real missing radio definitions** (the inventory agent
   found ~22 candidates; 8 were hallucinated; 14 confirmed real
   Hamlib radios in our active scope). The five highest-value
   additions:
   - **Yaesu FTDX-5000** (2010 HF 200W flagship, `yaesu/ft5000.c`)
   - **Yaesu FTDX-9000** (2007 HF/VHF/UHF flagship, `yaesu/ft9000.c`)
   - **Yaesu FT-2000** (2005 HF 200W, `yaesu/ft2000.c`)
   - **Icom IC-F8101** (HF marine/expedition, `icom/icf8101.c`) —

   - **Kenwood TH-F6A / TH-F7E** (popular tri-band HT,
     `kenwood/thf6a.c`)

4. **Two operator-facing capability gaps**:

### Defer to v1.3 / Phase 6

6. Function toggle gaps (CSQL, DSQL, BC2, etc.) — low demand.
7. Configuration parameters (BACKLIGHT, BEEP, TIME, AFIF).
8. Spectrum scope waveform streaming.
9. Transceive mode (async state push).
10. D-STAR call-sign / message operations.

## 2. Command-divergence findings (verified)

The divergence agent did a deep Icom scan and a sample of
Kenwood/Yaesu/Elecraft. Findings:

### Critical — fix in v1.1.1

**IC-9700 / IC-705 noise blanker BCD encoding bug.**

- **File**: `Sources/RigControl/Protocols/Icom/IcomCIVProtocol+NoiseControl.swift:143`.
- **Code**:
  ```swift
  let bcdLevel = [UInt8(level % 10) | (UInt8(level / 10) << 4), UInt8(level / 100)]
  ```
- **Bug**: for `level = 128`, produces `[0xC8, 0x01]`. The `0xC` nibble
  is not a valid BCD digit — Icom radios expect packed BCD with
  digits 0–9 per nibble. Symptom: radio NAKs or stores a garbage
  level.
- **Reachable via**: `setNoiseBlanker(.enabled(level: 128))` —
  the public API accepts 0–255.
- **History**: the IC-7100 path was fixed in commit `b2f2ca4`
  ("CRITICAL FIX: Correct BCD encoding for IC-7100 Level
  commands"). The fix introduced the `BCDEncoding.encodePower`
  helper. The IC-9700/IC-705 path was added later but inlined
  the broken math directly instead of calling the helper.
- **Fix**: one-line change — call `BCDEncoding.encodePower(Int(level))`
  matching the IC-7100 path at `IcomCIVProtocol+IC7100Controls.swift:157`.

### Verified clean (the agent confirmed Hamlib parity)

- Icom mode setting (including the IC-7600 DATA-USB fix from
  commit `a1b1396`).
- Icom PTT, frequency BCD, RIT/XIT offset encoding.
- Icom VFO ops, function toggles (the v1.1 surface).
- Icom scanning, CW keyer, IF filter (FIL1/2/3 selection logic).
- Icom AGC code mapping per radio.
- Signal strength S-unit calibration.
- Power-level percentage scaling.

### Audit scope caveat

The agent's deep dive was Icom-focused. Kenwood, Yaesu, and
Elecraft were sampled but not exhaustively walked. A follow-up
audit of those three vendor protocol surfaces would be sensible
before v1.2 ships — flagged as a v1.1.1 task below.

## 3. Capability-gap findings (verified)

The capability-gap agent re-ran the HAMLIB_PARITY.md audit with
fresh eyes after the v1.1 parity work. Most of its findings line
up with the existing Phase 6 list; the new identifications:

### Tier-1 capability gaps for v1.2

- **Per-mode IF filter selection** (FIL1/2/3 per mode). High
  value — CW operators routinely switch filters per band.
  Requires extending `RigCapabilities` with a `filtersPerMode`
  table and a new `SupportsFilterSelection` trait. Wire commands
  exist in Hamlib's `rig_caps.filters[]`.

- **Tuning step (TS) set/get**. `availableTuningSteps` is
  modeled in capabilities but has no setter. Adding it is a
  small trait protocol with Icom `0x1A 0x05`, Kenwood `TS`,
  Yaesu `ST`.

### Tier-2 capability gaps for v1.2 (optional)

- **Extended RigFunction enum** with CSQL, DSQL, BC2 — Hamlib
  bits that some flagship radios surface. Cheap to add (just
  more enum cases + per-vendor wire commands). The Hamlib
  function bit count is ~52; we shipped 21 in v1.1.

### Deferred — confirmed Phase 6

The agent confirmed the existing Phase 6 deferral list still
makes sense:

- Spectrum scope streaming (async transport refactor).
- Configuration parameters / parm trait (BACKLIGHT, BEEP, TIME,
  AFIF audio routing) — low demand.
- D-STAR call signs / messages.
- Clock setting (RIG_PARM_TIME).
- Transceive mode (async state push from radio).
- Calibration tables (`str_cal`, `swr_cal`, `rfpower_meter_cal`).
- AGC time constant (separate from speed).
- Antenna option arg (RX-only vs. TX routing).

### Reconsidered

The capability agent suggested promoting `RIG_FUNC_TRANSCEIVE`
and `RIG_FUNC_SPECTRUM` to v1.2. **I disagree** — both require
async transport plumbing that doesn't exist today. The polling
`events` stream we ship is the right abstraction for now; adding
true async-push without the transport layer to back it would
ship a no-op enum case. Defer until the transport refactor.

## 4. Documentation-audit findings (verified)

Heavy lift here. The documentation agent's findings are largely
on the money — the v1.1 features are real, working, tested, and
**effectively invisible** in user-facing docs.

### Blockers (must fix before next external announcement)

1. **README "Features" section doesn't mention v1.1 features.**
   Add a "What's new in v1.1" subsection with 3 one-line
   examples (VFO ops, function toggles, secondary levels).

2. **`Documentation/USAGE_EXAMPLES.md` has no v1.1 examples.**
   Add four sections:
   - VFO operations with all enum cases.
   - Function toggles with common bits.
   - Secondary level controls.
   - rigctld bridge with WSJT-X integration.

3. **DocC catalog missing five articles.** The hosted DocC site
   at `jjones9527.github.io/SwiftRigControl/documentation/rigcontrol/`
   is the canonical reference. Currently lacks articles on:
   - `VFOOperations.md`
   - `FunctionToggles.md`
   - `SecondaryLevels.md`
   - `TraitProtocols.md` (explaining the architecture)
   - `RigctldBridge.md`

4. **README's "Supported Radios" table is missing the seven new
   v1.1 radios** — ID-31, ID-51, ID-52, IC-92D, IC-R30, TH-D75,
   TX-500. They exist in the codebase but aren't advertised.

5. **Multiple docs reference "post-v2.0 deprecation removal"**
   for changes that actually shipped in Phase 1.4 of v1.0.x.
   `API_REFERENCE.md` and `USAGE_EXAMPLES.md` are the worst
   offenders. There is no v2.0; the docs need correcting.

### Polish gaps (v1.2 timeframe)

6. **No "Getting Started in 5 minutes" guide** for Swift
   developers integrating SwiftRigControl. Recommended new doc:
   `Documentation/GETTING_STARTED.md`.

7. **`Documentation/ADDING_RADIOS.md` doesn't cover trait
   protocols.** New contributors don't see the
   `Supports*`-protocol-conformance pattern from Phase 5.1.

8. **`CLAUDE.md` doesn't mention the `VendorExtensions` enum**
   pattern (Phase 5.2). Contributors using `as?` casts won't
   discover the typed dispatch.

9. **`TROUBLESHOOTING.md` should explain the IC-7600 DATA-USB
   `0x1A 0x06` sub-command quirk** — both for users debugging
   data-mode issues and for contributors hitting similar quirks
   on other radios.

10. **`NETWORK_CONTROL.md` should highlight the new v1.1 rigctld
    commands** (`vfo_op`, expanded `set_func`/`get_func`,
    secondary levels) and include a WSJT-X integration example.

### Nice-to-have

11. `SERIAL_PORT_GUIDE.md` flow-control + USB descriptor
    filtering subsection.

12. `XPC_HELPER_GUIDE.md` Mac App Store entitlements section.

13. Cross-link the contributor-facing `ADDING_RADIOS.md` from
    the DocC catalog landing page.

## 7. Confidence calibration

What I'm confident in:

- The IC-9700 NB BCD bug — verified the code, the math is
  demonstrably wrong, the fix matches an established pattern.
- The documentation drift — every claim is checkable in seconds.
- The Phase 6 deferral list — the gap agent's analysis aligns
  with what was already documented in HAMLIB_PARITY.md.

What I'm less certain about:

- The Tier-1 radio priorities. "FTDX-5000 and FT-2000 still
  have userbase" is my judgment; data on actual amateur radio
  market share is sparse. Recommend treating the radio list as
  "ship-when-asked" rather than mandatory.
- The capability agent's per-mode-filter and TS proposals are
  good ideas but neither has a user request behind it. They're
  defensible v1.2 additions but not urgent.

What I deliberately did NOT trust without verification:

- The radio inventory agent's invented model names (rejected 8).
- Anything from agents that didn't cite a file:line or commit
  hash.
- Quantitative claims about "20% gap" — replaced with
  conservative bounds.
