# v1.1.0 Hardware Test Plan

Pre-tag validation for the v1.1.0 release. Run these on the
machine that has the radios physically connected. Total wall-clock
time: ~20–30 minutes for all four radios; ~5 minutes per radio.

## Prerequisites

- macOS 14+ with Xcode 16+
- Repo cloned at the target machine: `git clone …; cd SwiftRigControl`
- One or more of: Icom IC-7100, IC-7600, IC-9700, Elecraft K2
- USB serial adapter(s) wired to each radio's CAT port

```bash
# Make sure you're on the v1.1 release tag (or the commit you intend to tag).
git log --oneline -1
swift build       # Sanity check: library compiles on this machine
```

If `swift build` fails on this machine but worked on the development
machine, **stop** — there's a portability bug we need to fix before
shipping.

## Step 0 — Identify your serial ports

Plug each radio in **one at a time** and run:

```bash
ls /dev/cu.* | grep -v Bluetooth
```

Note the new entry for each radio. Example (your paths will differ):

- IC-7100 → `/dev/cu.usbserial-AB0NZQ8K`
- IC-7600 → `/dev/cu.SLAB_USBtoUART`
- IC-9700 → `/dev/cu.usbmodem14101`
- K2     → `/dev/cu.usbserial-FT8XYZWY`

Export them as environment variables so the validators pick them up:

```bash
export IC7100_SERIAL_PORT="/dev/cu.usbserial-AB0NZQ8K"
export IC7600_SERIAL_PORT="/dev/cu.SLAB_USBtoUART"
export IC9700_SERIAL_PORT="/dev/cu.usbmodem14101"
export K2_SERIAL_PORT="/dev/cu.usbserial-FT8XYZWY"
```

## Step 1 — Validate the existing hardware suites

These four validators were green at v1.0.6 and exercise the bulk of
the CATProtocol surface. The v1.1 namespace move touched their call
sites; we need to confirm none of those edits regressed behavior on
the wire.

For each radio you have:

```bash
cd Tools/SwiftRigControlTools
swift run IC7600Validator    # or IC7100Validator / IC9700Validator / K2Validator
```

**Pass criteria** — exit code 0 plus a "All tests passed" banner.
If any test fails, capture the full stdout, note which test failed,
and stop before tagging.

What each validator covers (the ones I'd flag if they fail):

| Test                | Why it matters for v1.1                                       |
| ------------------- | ------------------------------------------------------------- |
| Frequency set/get   | Sanity — confirms the namespaced `.Icom.icXxxx()` factory built the right protocol with the right civAddress. |
| Mode control        | Covers the mode-encoding paths that were also touched by IC-7600 DATA-USB and IC-7100/IC-705 DATA-mode fixes from v1.0.6; v1.1 didn't change them, so any regression here is a portability issue, not a v1.1 issue. |
| Dual VFO / Split    | Confirms VFO selection still routes correctly post-namespace. |
| Power control       | Confirms the power-units regression-prone path is intact.    |
| PTT                 | Critical — if PTT doesn't release, you have a problem.       |
| Signal strength     | Quick sanity, low risk.                                      |
| RIT                 | RIT is heavily traited; touched by the Phase-5 capability split. |

## Step 2 — Validate `RadioDiscovery` against each radio

This is **new code that has never seen real hardware**. The TCPSerialTransport
and Flex definitions ship without hardware verification (no Flex on hand);
auto-detection is the one piece of v1.1 we *can* validate.

Run the discovery validator I added to the Tools tree:

```bash
cd Tools/SwiftRigControlTools

# With one radio connected at a time:
swift run RadioDiscoveryValidator
```

The validator iterates the radios you've exported port env vars for,
calls `RadioDiscovery.detect()` for each, and reports:

- Whether the radio was found on any port
- Whether the matched port equals the one you set in `*_SERIAL_PORT`
- The raw identity response

**Pass criteria** — every radio whose `*_SERIAL_PORT` env var is set
gets matched on that exact port. If it matches a *different* port
(another radio's port or a Bluetooth adapter), that's a real bug.
If it returns `nil`, the per-vendor identify probe needs adjustment.

### Multi-radio test (optional, only if you can connect >1 at once)

If you have two or more radios connected simultaneously, the
validator runs the multi-radio overload at the end and confirms each
appears exactly once in the result list. This exercises the
port-exclusivity logic.

## Step 3 — Smoke test the rigctld bridge

The rigctld bridge is unchanged in v1.1 except for one capability
flag (Flex), but it's worth a 30-second sanity check before tagging.

```bash
# Terminal A — bring up the bridge against any verified radio:
cd Tools/SwiftRigControlTools
IC7600_SERIAL_PORT="/dev/cu.XXX" swift run RigctldEmulator IC7600
```

```bash
# Terminal B — drive it with a Hamlib client (if installed) or netcat:
echo "f" | nc localhost 4532       # Should return the current frequency
echo "m" | nc localhost 4532       # Should return current mode
echo "q" | nc localhost 4532       # Quit
```

**Pass criteria** — `f` and `m` return well-formed Hamlib responses.

## Step 4 — Capture a report

After the runs are done, capture a one-paragraph summary you can
paste into the GitHub release notes:

> Validated v1.1.0 on real hardware: IC-7100 / IC-7600 / IC-9700 /
> K2. All four hardware validators pass; RadioDiscovery correctly
> identifies each radio at its expected serial port and baud rate;
> rigctld bridge responds correctly to `f` and `m`.

…or, if anything failed, file a GitHub issue with the failing
output before tagging.

## When to tag

Tag v1.1.0 *only* after:

1. Step 1 (existing validators) passes on at least one Icom radio
   and the K2.
2. Step 2 (discovery) passes on every radio you tested.
3. CHANGELOG `[1.1.0]` heading is in place (already done).
4. `Documentation/MIGRATION_v1.1.md` is published (already done).

```bash
git tag -a v1.1.0 -m "v1.1.0: vendor namespaces, TCP transport, Flex, auto-detection"
git push origin v1.1.0
```

The GitHub Actions workflow will deploy the updated DocC site
automatically.
